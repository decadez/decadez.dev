# AGENTS.md

## Project Overview

`decadez.dev` is a pnpm monorepo for the Decadez personal site, API, documentation site, and reusable packages.

- Package manager: `pnpm@11.7.0`
- Node.js version: `22`
- Main frontend stack: Next.js, React, TypeScript, Tailwind CSS
- API stack: NestJS, TypeScript
- Shared packages: UI components and Nest infrastructure utilities

## Development Commands

Run commands from the repository root unless noted otherwise.

```bash
pnpm install
pnpm dev:web
pnpm dev:docs
pnpm dev:api
pnpm lint
pnpm format:check
pnpm format
pnpm build:web
pnpm build:docs
pnpm build
```

Workspace-specific commands:

```bash
pnpm --filter @decadez/web dev
pnpm --filter @decadez/docs dev
pnpm --filter @decadez/api start:dev
pnpm --filter @decadez/web-dev-ui lint
pnpm --filter @decadez/nest-infra build
```

Publishing commands:

```bash
pnpm publish:web-dev-ui
pnpm publish:nest-infra
```

## Project Structure

```text
apps/
  web/      Next.js personal site and blog
  docs/     Next.js documentation site for shared UI/MDX components
  api/      NestJS API service
packages/
  web-dev-ui/   Shared React UI and MDX/Sandpack components
  nest-infra/   Shared NestJS infrastructure providers
```

Important root files:

- `package.json` defines monorepo scripts.
- `pnpm-workspace.yaml` defines workspace packages under `apps/*` and `packages/*`.
- `.prettierrc` defines repository formatting.
- `pnpm-lock.yaml` is generated by pnpm and should not be hand-edited.

## Code Conventions

- Use TypeScript for application and package code.
- Prefer existing workspace packages and local helpers over introducing duplicate implementations.
- Keep shared UI behavior in `packages/web-dev-ui`; docs should consume it instead of reimplementing component logic.
- Keep shared NestJS infrastructure in `packages/nest-infra`; app-specific API modules belong in `apps/api`.
- Use Prettier defaults from `.prettierrc`: double quotes, semicolons, 2-space indentation, trailing commas where valid in ES5, and 80-character print width.
- Do not commit generated build output such as `.next`, `out`, `dist`, or coverage artifacts unless explicitly required.
- Preserve workspace dependency links with `workspace:*` for local packages.

## Environment

Local env files are ignored by git. Use `.env.local` in the relevant app when needed.

Common frontend variables:

- `NEXT_PUBLIC_URL`
- `NEXT_PUBLIC_BASE_PATH`
- `NEXT_PUBLIC_DOCS_BASE_PATH`
- `NEXT_PUBLIC_EMAIL`
- `NEXT_PUBLIC_API_BASE_URL`
- `GITHUB_PAGES`

API and infrastructure variables:

- `PORT`
- `CORS_ORIGIN`
- `REDIS_URL`
- `EDGE_CONFIG`
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`

Do not commit secrets or real local environment values.

## Validation

Before handing off code changes, run the narrowest relevant check first:

```bash
pnpm --filter @decadez/web lint
pnpm --filter @decadez/docs lint
pnpm --filter @decadez/api lint
pnpm --filter @decadez/web-dev-ui lint
pnpm --filter @decadez/nest-infra lint
```

For broad changes, run:

```bash
pnpm lint
pnpm format:check
```

## Git Workflow

- Start new work from `main` or `master`, pull the latest content, then create a feature branch for the request.
- Use Angular-style commit messages.
- After pushing a feature branch, merge it into `dev/YYYYMM` for the current year and month. If that branch does not exist, create it from the latest `main` or `master`, push it, then return to the feature branch.
